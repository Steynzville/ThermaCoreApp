# Password Hash Column Migration

## Overview
This migration changes the `password_hash` column in the `users` table from `VARCHAR(128)` to `TEXT` to accommodate longer password hashes generated by modern hashing algorithms like scrypt.

## Why This Change?
Modern password hashing algorithms like scrypt can generate hashes that are longer than 128 characters. For example:
- **pbkdf2:sha256**: ~80-100 characters
- **scrypt**: 150-200+ characters
- **argon2**: 90-120 characters

The previous `VARCHAR(128)` limit could cause "value too long for type character varying(128)" errors when using scrypt or other modern algorithms.

## Migration Files

### 005_fix_password_hash_length.sql
This is the migration script that alters the existing `password_hash` column from `VARCHAR(128)` to `TEXT`.

**For PostgreSQL:**
```sql
ALTER TABLE users ALTER COLUMN password_hash TYPE TEXT;
ALTER TABLE users ALTER COLUMN password_hash SET NOT NULL;
```

**For SQLite:**
SQLite doesn't support ALTER COLUMN TYPE directly, but TEXT is already compatible with VARCHAR. If you need to migrate an existing SQLite database, you can skip this migration as SQLite's type affinity system will handle it automatically.

## How to Apply

### For New Installations
If you're setting up the database from scratch:
1. Use the updated schema files (`001_initial_schema.sql` or `001_initial_schema_sqlite.sql`)
2. These already have `password_hash` defined as `TEXT NOT NULL`

### For Existing Databases

#### PostgreSQL
```bash
# Connect to your database
psql -U <username> -d <database_name>

# Run the migration
\i backend/migrations/005_fix_password_hash_length.sql

# Or via Flask CLI (if you have custom commands)
flask db upgrade
```

#### SQLite
SQLite automatically handles the type conversion due to its type affinity system. No migration needed, but you should update your model definition.

### Verification
After applying the migration, verify the change:

**PostgreSQL:**
```sql
SELECT column_name, data_type, character_maximum_length 
FROM information_schema.columns 
WHERE table_name = 'users' AND column_name = 'password_hash';
```

Expected result: `data_type` should be `text`

**SQLite:**
```sql
PRAGMA table_info(users);
```

Look for the `password_hash` row - type should be `TEXT`

## Impact
- **Downtime**: Minimal (ALTER COLUMN is typically fast)
- **Data Loss**: None - existing data is preserved
- **Rollback**: Can revert to VARCHAR(255) if needed, but VARCHAR(128) would truncate data
- **Performance**: No impact - TEXT performs similarly to VARCHAR in PostgreSQL

## Troubleshooting

### "value too long for type character varying(128)"
This error means you need to apply this migration. Follow the steps above.

### "column password_hash cannot be cast automatically to type text"
This shouldn't happen, but if it does:
```sql
ALTER TABLE users ALTER COLUMN password_hash TYPE TEXT USING password_hash::TEXT;
```

### Existing users can't log in after migration
The migration preserves all existing password hashes. If users can't log in:
1. Check that the migration completed successfully
2. Verify password_hash column contains data: `SELECT id, username, length(password_hash) FROM users;`
3. Test with a known-good user/password combination
